## 一、可行性结论

**完全可行，技术成熟，方案丰富。**

网页端 3D 魔方已有大量成熟实现（Three.js + cubejs 是黄金组合），多人联机也有现成的 WebRTC/WebSocket 方案。核心难点不在技术实现，而在于体验打磨和功能差异化。

## 三、3D 渲染方案对比

| 方案 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **Three.js (原生)** | 完全控制、轻量、社区庞大 | 需手动处理交互 | ⭐⭐⭐⭐⭐ |
| **React Three Fiber** | React 生态、声明式、Drei 辅助 | 额外依赖、React 耦合 | ⭐⭐⭐⭐ |
| **CSS 3D Transforms** | 零依赖、CSS 即可 | 性能差、交互受限 | ⭐⭐ |
| **Raw WebGL** | 极致性能、完全掌控 | 开发成本极高 | ⭐ |
| **cubing.js / Twizzle** | 专业魔方库、支持多种魔方 | 定制受限 | ⭐⭐⭐⭐ |

### 推荐选择

- **方案 A（推荐）**: Three.js + 原生 JS/TS，无框架依赖，完全可控
- **方案 B**: React Three Fiber + Drei，如果项目用 React
- **方案 C**: cubing.js 的 TwistyPlayer 作为基础，在其上扩展游戏逻辑

### 3D 魔方核心实现要点

1. **Cubelet 模型**：27 个小立方体（3×3×3），每个有位置坐标 + 6 面颜色
2. **旋转系统**：临时将旋转层放入 THREE.Group → 动画旋转 → 重新附加到场景
3. **状态同步**：逻辑状态（solver 用）与 3D 渲染状态必须双向同步
4. **交互**：鼠标拖拽旋转面 + 键盘快捷键（R L U D F B + Shift 反转）

## 四、魔方求解器方案

| 算法 | 步数 | 速度 | 实现难度 | 适用场景 |
|------|------|------|----------|----------|
| **Kociemba 两阶段** | ≤20 步（最优） | 0.01-0.4s | 中 | 自动求解、提示 |
| **CFOP** | ~55 步 | 快 | 中 | 人类教学法 |
| **Beginner Method** | ~100 步 | 快 | 低 | 新手教程 |
| **Thistlethwaite** | ≤45 步 | 中 | 高 | 理论方案 |

### 推荐库

- **cubejs** (ldez/cubejs) — Kociemba 两阶段，354 stars，MIT 协议，支持 Web Worker
- **cubing.js** — 专业魔方库，支持多种魔方（2×2 到 7×7），内置 TwistyPlayer
- **rubiks-cube-solver** (npm) — CFOP 实现，简单易用

### Web Worker 隔离

Kociemba 的 pruning table 初始化需要 3-5 秒，求解偶尔达 1-2 秒，**必须放入 Web Worker** 避免阻塞 UI。

## 五、多人联机方案

### 5.0 局域网部署策略

本项目**优先支持局域网（LAN）模式**，这是区别于竞品的核心特性：

| 特性 | 说明 |
|------|------|
| **零部署** | 自己的电脑即服务器，`localhost:3000` 即可访问 |
| **超低延迟** | 局域网内 < 5ms，远优于公网 |
| **内网穿透** | 通过 ngrok / frp / cloudflared 等工具暴露给外网朋友 |
| **无需域名** | 局域网内 HTTP 即可运行，WebRTC 也支持 |
| **隐私优先** | 游戏数据不经过第三方服务器 |

**内网穿透方案对比**：

| 工具 | 难度 | 延迟 | 免费额度 | 推荐度 |
|------|------|------|----------|--------|
| **ngrok** | 极低 | 低 | 1 隧道 | ⭐⭐⭐⭐⭐ |
| **frp** | 中 | 最低 | 免费（自部署） | ⭐⭐⭐⭐ |
| **cloudflared** | 低 | 低 | 免费 | ⭐⭐⭐⭐ |
| **ZeroTier** | 低 | 低 | 25 设备 | ⭐⭐⭐⭐ |

### 5.1 现有竞品分析

| 产品 | 模式 | 特点 |
|------|------|------|
| **SpeedCubeBattle.com** | 1v1 对战 | OLL/PLL 训练、排行榜 |
| **mcubesarena.com** | 1v1 + 团队 + 淘汰赛 | 12 种对战模式 |
| **Plus2.me** | 实时竞速 | ELO 段位制 |
| **Speed (ryanheise.com)** | 1v1 对战 | 经典实现 |
| **acubemy.com** | 1v1 | 智能魔方联动 |

### 5.2 网络传输协议对比

| 协议 | 延迟 | 可靠性 | 复杂度 | 适用 |
|------|------|--------|--------|------|
| **WebSocket** | 100-300ms | TCP 可靠 | 低 | 房间、聊天、匹配 |
| **WebRTC DataChannel** | 30-80ms | 可配置 | 高 | 实时操作同步 |
| **WebTransport** | 最低 | 可配置 | 中 | 未来最佳选择 |

### 5.3 推荐架构：混合模式

```
匹配/房间管理 ──── WebSocket (Socket.IO) ──── 可靠有序
实时操作同步 ──── WebRTC DataChannel ──── 低延迟 P2P
信令协商 ──── WebSocket ──── 必须
```

**理由**：
- 魔方对战不是 FPS，不需要 60fps 同步
- 关键操作（转动）需要可靠到达，但延迟要低
- WebRTC 的 unreliable mode 适合实时状态
- WebSocket 做房间管理最简

### 5.4 同步策略

| 策略 | 说明 |
|------|------|
| **Server-Authoritative** | 服务器验证所有操作，防作弊（推荐用于竞技） |
| **P2P + 服务器仲裁** | 操作 P2P 发送，服务器做最终判定 |
| **Lockstep** | 锁定步调，双方确认后推进（适合回合制） |

### 5.5 游戏模式

#### 模式 A：1v1 对战（竞速）

```
Client A ──join──► Server ◄──join── Client B
                    │
                    ▼
              Room Created
              ├─ scramble (服务器生成，保证一致)
              ├─ player states
              ├─ timer
              └─ result

操作流:
Player A 转动 ──► 本地立即渲染 + 发送操作
                    ▼
              WebRTC / WS ──► Player B
                    ▼
              Server 验证 ──► 广播确认
```

#### 模式 B：协作解魔方（Co-op）

```
Player A ──join──► Server ◄──join── Player B
                    │                    │
                    ▼                    ▼
              Room Created          Player C ──join──┐
              ├─ scramble (服务器生成，保证一致)        │
              ├─ shared cube state ◄───────────────┘
              ├─ turn system (轮流 or 自由)
              └─ result

协作规则:
- 所有玩家看到同一个魔方
- 可选轮流模式（每人一步）或自由模式（谁都能转）
- 所有操作实时同步给所有玩家
- 计时器统一计算，团队成绩
- 支持 2-4 人协作
```

#### 模式 C：房间系统通用设计

```
Room
├─ id: string (6位字母数字)
├─ mode: '1v1' | 'coop' | 'practice'
├─ host: playerId
├─ players: Player[]
├─ spectators: Player[]
├─ maxPlayers: 2 (1v1) / 4 (coop)
├─ scramble: string | null
├─ status: 'waiting' | 'playing' | 'finished'
├─ createdAt: timestamp
└─ settings: {
    hintEnabled: boolean,  // 仅练习模式可用
    timeLimit: number | null,
    chatEnabled: boolean
  }
```

### 7.4 技术实现

```typescript
// HintEngine.ts
class HintEngine {
  private solver: Worker;
  
  async getHint(currentState: CubeState, difficulty: 'easy' | 'medium' | 'hard'): Promise<Hint> {
    // 调用 Kociemba 求解器
    const solution = await this.solver.solve(currentState);
    const moves = solution.split(' ');
    const nextMove = moves[0]; // 最优下一步
    
    return {
      move: nextMove,           // e.g. "R"
      layer: parseLayer(nextMove), // e.g. "R"
      direction: parseDirection(nextMove), // "clockwise" | "counterclockwise"
      description: formatHint(nextMove), // "右面顺时针旋转"
      // 高亮目标层的所有 cubelet
      highlightCubies: getCubiesForLayer(nextMove),
    };
  }
}

// 提示状态管理 (Zustand)
interface HintState {
  currentHint: Hint | null;
  hintLevel: 1 | 2 | 3;
  hintsUsed: number;
  maxHints: number; // 限制使用次数
  showHint: () => void;
  clearHint: () => void;
}
```

### 7.5 防作弊机制

```typescript
// 多人模式中禁用提示
function canUseHint(mode: GameMode): boolean {
  return mode === 'practice'; // 仅练习模式可用
}

// 服务端验证
socket.on('request-hint', (data) => {
  if (room.mode !== 'practice') {
    socket.emit('hint-denied', '提示功能仅练习模式可用');
    return;
  }
  if (player.hintsUsed >= room.maxHints) {
    socket.emit('hint-denied', '提示次数已用完');
    return;
  }
  // 发送提示给请求者（不广播）
  const hint = generateHint(data.cubeState);
  socket.emit('hint', hint);
});
```

| 方案 | 工作量 | 体验 | 推荐度 |
|------|--------|------|--------|

| **C. 保持网页版 + 小程序 WebView** | 低 | 中 | ⭐⭐⭐⭐ |

### 8.2 推荐方案：Taro 跨端框架

```
┌─────────────────────────────────────────┐
│              Taro 跨端框架               │
│                                         │
│  一套代码 ──┬──► Web (H5)              │
│             ├──► 微信小程序              │
│             ├──► 支付宝小程序            │
│             └──► React Native (App)     │
└─────────────────────────────────────────┘
```

### 8.3 迁移要点

| 模块 | 迁移策略 |
|------|----------|
| **3D 渲染** | Three.js → 小程序 WebGL canvas（需适配） |
| **网络** | Socket.IO → 小程序 WebSocket / wx.connectSocket |
| **状态管理** | Zustand → Taro 内置或 Mobx |
| **Worker** | Web Worker → 小程序不支持，改用云函数或主线程 |
| **存储** | localStorage → wx.setStorageSync |

### 8.4 小程序限制与应对

| 限制 | 应对方案 |
|------|----------|
| 不支持 Web Worker | 求解器放云函数或分包异步加载 |
| 包体积限制 (2MB) | 代码分包，3D 资源 CDN 加载 |
| WebGL 性能限制 | 降低渲染质量，关闭特效 |
| Socket 限制 | 使用小程序 WebSocket API |

### 11.2 Three.js 旋转层实现

```typescript
function rotateLayer(cubies: THREE.Mesh[], axis: 'x' | 'y' | 'z', layer: number, angle: number) {
  const group = new THREE.Group();
  
  // 将属于该层的小立方体放入临时组
  cubies.forEach(cubie => {
    const pos = cubie.userData.logicalPosition;
    if (pos[axis] === layer) {
      group.add(cubie);
    }
  });
  
  // 动画旋转
  animate(group, { axis, angle, duration: 200 }, () => {
    // 动画完成后重新附加到场景
    scene.attach(group);
    // 更新 logicalPosition
    updateLogicalPositions(cubies);
  });
}
```

### 11.3 WebRTC P2P 连接

```typescript
class P2PManager {
  private peerConnection: RTCPeerConnection;
  private dataChannel: RTCDataChannel;
  
  async connect(peerId: string) {
    this.peerConnection = new RTCPeerConnection({
      iceServers: [{ urls: 'stun:stun.l.google.com:19302' }]
    });
    
    this.dataChannel = this.peerConnection.createDataChannel('cube-sync', {
      ordered: false,  // 允许乱序，降低延迟
      maxRetransmits: 0  // 不重传，追求最低延迟
    });
    
    this.dataChannel.onmessage = (e) => {
      const move = JSON.parse(e.data);
      this.onRemoteMove(move);
    };
    
    // 通过 Socket.IO 交换 SDP
    const offer = await this.peerConnection.createOffer();
    await this.peerConnection.setLocalDescription(offer);
    socket.emit('webrtc-offer', { to: peerId, offer });
  }
  
  sendMove(move: Move) {
    if (this.dataChannel?.readyState === 'open') {
      this.dataChannel.send(JSON.stringify(move));
    }
  }
}
```

### 10.4 Socket.IO 房间管理

```typescript
// Server
io.on('connection', (socket) => {
  socket.on('create-room', (config) => {
    const roomId = generateRoomId();
    socket.join(roomId);
    rooms.set(roomId, { 
      host: socket.id, 
      players: [socket.id],
      mode: config.mode || '1v1',  // '1v1' | 'coop' | 'practice'
      hintEnabled: config.mode === 'practice',
      maxHints: 3
    });
    socket.emit('room-created', roomId);
  });
  
  socket.on('join-room', (roomId) => {
    const room = rooms.get(roomId);
    if (!room) return socket.emit('error', 'Room not found');
    if (room.players.length >= room.maxPlayers) return socket.emit('error', 'Room full');
    
    socket.join(roomId);
    room.players.push(socket.id);
    io.to(roomId).emit('player-joined', socket.id);
    
    // 人满后自动开始
    if (room.players.length === room.maxPlayers) {
      const scramble = generateScramble();
      io.to(roomId).emit('game-start', { scramble, mode: room.mode });
    }
  });
  
  socket.on('move', (data) => {
    const room = rooms.get(data.roomId);
    if (!room) return;
    if (validateMove(data.move)) {
      socket.to(data.roomId).emit('opponent-move', data.move);
    }
  });
  
  // 协作模式：广播给所有人（包括发送者，用于确认）
  socket.on('coop-move', (data) => {
    const room = rooms.get(data.roomId);
    if (!room || room.mode !== 'coop') return;
    if (validateMove(data.move)) {
      io.to(data.roomId).emit('cube-update', data.move); // 广播给所有人
    }
  });
});
```

```typescript
// HintEngine.ts
class HintEngine {
  private solver: Worker;
  
  async getHint(currentState: CubeState): Promise<Hint> {
    const solution = await this.solver.solve(currentState);
    const moves = solution.split(' ');
    const nextMove = moves[0];
    
    return {
      move: nextMove,
      layer: parseLayer(nextMove),      // e.g. "R"
      direction: parseDirection(nextMove), // "clockwise" | "counterclockwise"
      description: formatHint(nextMove),  // "右面顺时针旋转"
      highlightCubies: getCubiesForLayer(nextMove),
    };
  }
}
```

## 十三、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| WebRTC NAT 穿透失败 | P2P 连不上 | 部署 TURN 服务器做中继 |
| 求解器初始化慢 | 首次加载 3-5 秒 | Web Worker + loading 提示 |
| 状态不同步 | 双方看到不同魔方 | 服务器定期广播完整状态校验 |
| 作弊 | 竞技公平性 | 服务器验证 + 操作时间戳校验 |
| 移动端性能 | 卡顿 | 降低渲染质量 + 关闭特效 |
| 提示系统被滥用 | 多人模式不公平 | 仅练习模式可用 + 次数限制 |
| 小程序包体积超限 | 无法发布 | 代码分包 + 资源 CDN |
| 局域网穿透失败 | 外网朋友无法访问 | 提供多种穿透工具备选 |
