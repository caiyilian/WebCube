# WebCube 剩余功能补完计划

> 基于 2026-06-28 对照 `方案.md` 的构建、单测、浏览器手测和代码巡检结果整理。本文只记录后续实现计划，不代表功能已经完成。

---

## 一、当前验收结论

项目已经具备较完整的基础骨架：

- Web 端可以启动并渲染 3D 魔方。
- `npm run build` 通过。
- `npm run test:run` 通过，当前 5 个测试文件、54 个用例通过。
- 前后端服务都能启动，后端健康检查、房间列表、排行榜、创建房间接口有响应。
- 代码中已经存在多人房间、匹配、锦标赛、AI、CFOP、回放、主题、音效、Taro、RN 等模块雏形。

但对照 `方案.md` 的五个阶段，当前还不能认定为全部完成。主要问题集中在：

- 多数高级功能只停留在底层类或服务端事件，没有接入用户可操作 UI。
- 前端模式、共享类型和后端模式命名不一致，例如 `battle` vs `1v1`。
- 服务端多人状态同步仍有占位实现，例如 `applyScramble()` 返回 solved state，`applyMoveToState()` 原样返回 state。
- 自动求解、提示系统、4x4 逻辑、Undo/Redo 显示、跨端构建等仍未达到方案验收标准。

## 二、阶段 0：验收阻塞与一致性修复

目标：先把当前已经暴露的明显错误修掉，保证后续功能开发不在错误基础上叠加。

### 0.1 清理首页与模式命名

验收目标：

- 首页只保留一组阶数选择。
- 只展示当前真实支持的阶数：2x2、3x3、4x4。
- 移除或禁用 5x5 入口，除非后续真正实现。
- 首页交互文案更新为当前真实操作方式：点击贴纸后使用方向键/WASD。
- 前端 `battle` 与共享类型/服务端 `1v1` 统一命名。

涉及文件：

- `src/components/HomePage.ts`
- `src/App.tsx`
- `shared/types.ts`
- `server/index.ts`
- `server/rooms/RoomManager.ts`

### 0.2 修复 Undo/Redo 用户可见状态

当前问题：

- Undo/Redo 后内部 `moveHistoryIndex` 会变化，但 HUD 显示仍使用 `moveHistory.length`，所以步数不变。

验收目标：

- Undo 后 HUD 步数减少。
- Redo 后 HUD 步数增加。
- 魔方渲染状态与逻辑状态一致。
- Undo/Redo 快捷键与按钮都可用。

### 0.3 修复调试暴露与运行时状态检查

当前问题：

- 浏览器里 `window.__WEBCUBE__` 在部分页面检查时不可读或未稳定暴露。

验收目标：

- 开发模式下可以稳定读取 `window.__WEBCUBE__.canvas`、`hud`、`settings`、`store`。
- 不影响生产构建。

### 0.4 清理重复/未接入组件

当前问题：

- `HUD.ts` 与 `HUD.tsx` 同时存在，实际入口使用较简单版本，容易造成维护混乱。

验收目标：

- 明确保留一个 HUD 实现，或将两者拆成明确职责。
- 删除或标记未使用组件。
- 设置面板、提示显示、操作按钮都接入同一个 HUD。

## 三、阶段 1：补齐单机核心功能

目标：让 Phase 1 真正达到 `方案.md` 里的完整单机 MVP。

### 1.1 求解器 Worker 接入

当前问题：

- 存在 `workers/solver.worker.ts`，但主流程中的自动求解不是 Worker 求解。
- `autoSolve()` 当前是直接 reset 到 solved state。

验收目标：

- Worker 初始化成功。
- 当前魔方状态能转换成 solver 输入。
- 调用 solve 后返回合法解法序列。
- Worker 初始化和求解过程不阻塞 UI。
- 求解失败时有明确错误提示。

### 1.2 自动求解动画

当前问题：

- 点击求解直接还原，没有按步骤动画。

验收目标：

- 点击求解后按求解步骤逐步旋转。
- HUD 显示求解中状态。
- 求解中禁用冲突操作，例如打乱、手动旋转、再次求解。
- 求解完成后状态为 solved。

### 1.3 提示系统完整化

当前问题：

- 提示主要是反向上一步或默认提示。
- UI 只有层高亮，没有完整文字描述和方向箭头。

验收目标：

- 提示来自 solver 的下一步。
- Level 1：只高亮目标层。
- Level 2：高亮 + 方向提示。
- Level 3：高亮 + 方向 + 文字描述。
- HUD 或提示面板显示中文提示，例如“右面顺时针旋转”。
- 多人/协作模式隐藏或禁用提示。

### 1.4 2x2、3x3、4x4 逻辑一致性

当前问题：

- 2x2 有部分状态逻辑。
- 3x3 相对完整。
- 4x4 目前只旋转本面，未完整更新相邻层。

验收目标：

- 2x2、3x3、4x4 的渲染、打乱、重置、手动旋转、Undo/Redo 都一致可用。
- 不支持求解的阶数要明确禁用求解按钮或展示说明。
- 4x4 若暂不接 solver，应明确标记“求解暂不可用”。

### 1.5 移动端触摸验收

验收目标：

- 手机视口下布局不溢出。
- 可点击贴纸并使用触摸操作旋转视角。
- HUD 按钮不遮挡主要画面。
- 触摸选择和拖拽视角不会互相误触。

## 四、阶段 2：多人联机可用化

目标：把已有服务端房间、匹配、聊天等能力接到前端，形成可玩的 1v1 联机流程。

### 2.1 前端网络客户端

验收目标：

- 新增或完善 Socket.IO 客户端封装。
- 支持连接、断开、重连、事件订阅和错误提示。
- 前端能显示当前连接状态。

建议模块：

- `src/net/RoomClient.ts`
- `src/stores/useRoomStore.ts`

### 2.2 房间 UI

验收目标：

- 创建房间，显示 6 位房间码。
- 输入房间码加入。
- 离开房间。
- 显示玩家列表、房主标识、准备状态。
- 房间错误有可见提示。

### 2.3 1v1 对战流程

当前问题：

- 后端有事件雏形，但前端没有 1v1 房间界面。
- 服务端状态更新仍有占位实现。

验收目标：

- 两个浏览器窗口可进入同一个房间。
- 双方准备后同时开始。
- 使用同一 scramble。
- 玩家操作后对方能看到同步。
- 先还原者获胜，显示结果。
- 胜负更新 ELO 和个人统计。

### 2.4 服务端状态逻辑补完

当前问题：

- `RoomManager.applyScramble()` 返回 solved state。
- `RoomManager.applyMoveToState()` 原样返回 state。

验收目标：

- 服务端能真实应用 scramble。
- 服务端能真实应用 move。
- 服务端能正确判断 solved。
- 服务端状态与客户端定期校验。

### 2.5 匹配系统接入

验收目标：

- 前端提供“随机匹配”按钮。
- 匹配中可取消。
- 两名玩家匹配成功后自动进入房间。
- 匹配失败或超时有提示。

### 2.6 聊天、排行榜、个人统计

验收目标：

- 房间内聊天实时广播。
- 排行榜页面显示 ELO 排名。
- 个人统计显示 best、average、历史记录、胜率。
- 数据至少在服务端运行期间持久存在；若要长期保存，需要再引入持久化方案。

## 五、阶段 3：协作模式可用化

目标：让 2-4 人协作解一个共享魔方成为可操作产品流程。

### 3.1 协作房间 UI

验收目标：

- 首页选择协作模式后进入协作房间流程。
- 支持创建房间和加入房间。
- 显示 2-4 名玩家。
- 房主可以开始游戏。

### 3.2 共享魔方状态同步

验收目标：

- 所有人看到同一个魔方。
- 任意玩家操作后所有人同步。
- 新加入观战者能收到当前完整状态。

### 3.3 轮流/自由模式

验收目标：

- 房主可切换轮流模式。
- 轮流模式显示当前轮到谁。
- 非当前玩家操作被前端禁用，并被服务端拒绝。
- 自由模式下任何玩家可操作。

### 3.4 团队计时与结果统计

验收目标：

- 协作模式统一计时。
- 还原后显示团队总时间。
- 显示每个玩家操作次数。
- 提示按钮在协作模式隐藏或禁用，服务端 API 也拒绝提示请求。

### 4.5 锦标赛模式

当前状态：

验收目标：

- 可创建 4/8/16 人锦标赛。
- 显示 bracket。
- 每轮匹配房间。

## 七、阶段 7：Electron 桌面应用打包

目标：将 WebCube 打包为跨平台桌面应用（Windows / macOS / Linux），用户可双击运行，无需额外搭建 Node 环境。

### 7.1 Electron 主进程

验收目标：

- 创建 `electron/main.ts` 主进程入口。
- 主进程启动 Express 后端（内嵌）或代理到已有后端。
- 创建 BrowserWindow，加载 Vite 构建产物。
- 窗口标题、图标、最小尺寸配置正确。
- 开发模式下加载 `localhost:5173`，生产模式加载 `dist/`。

### 7.2 预加载脚本与 IPC

验收目标：

- 预加载脚本暴露安全的 IPC 桥接。
- 主进程与渲染进程通信正常（例如获取版本号、平台信息）。
- 文件对话框、原生菜单等桌面能力通过 IPC 暴露。

### 7.3 打包配置

验收目标：

- 使用 `electron-builder` 配置打包。
- Windows 平台：NSIS 安装包（含自动更新基础）。
- macOS 平台：dmg 镜像。
- Linux 平台：AppImage / deb。
- 应用图标（至少 256×256）存在且正确引用。
- 打包产物 < 200MB（含 Node.js 运行时）。

### 7.4 后端内嵌模式

验收目标：

- 桌面版可选择内嵌后端模式（单进程，Electron 自启 Server）。
- 也可连接已有局域网/公网后端（多进程模式）。
- 模式切换在设置面板中可选。

### 7.5 桌面专属功能

验收目标：

- 系统托盘图标，支持最小化到托盘。
- 全局快捷键（如 Ctrl+Shift+R 重置魔方）。
- 原生菜单栏（File / Edit / View / Help）。
- 窗口大小可调整，最小尺寸 800×600。
- 退出时确认（游戏中防误关）。

8. 阶段 7：Electron 桌面应用打包（等 Web 端稳定后再做）。
